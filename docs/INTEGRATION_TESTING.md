# Integration Testing — Proof of Value

> Proving Sentinel catches real behavioral regressions, not just mock-on-mock.

## The Gap

The original Sentinel build (Phases 1-10) tested everything with mock agents
and mock tools. The adapters existed but hadn't been exercised against real
LangChain tool classes. This document records the integration testing that
fills that gap.

## What Was Tested

### 1. Real LangChain Tools Through Sentinel Adapters

**File:** `tests/sentinel/test_integration_langchain.py::TestRealToolIntegration`

We created real `@tool`-decorated functions (web_search, calculator, send_email)
with actual `name`, `description`, `args_schema`, and `invoke()` methods.
These went through sentinel's `SentinelToolAdapter` to verify:

- Adapter preserves real tool metadata (name, description, args schema)
- Adapter delegates execution to the mock (not the real tool)
- Multiple real tools share a single AgentTrace
- Type validation rejects non-BaseTool objects

**Result:** 5/5 tests pass. The adapter layer works correctly with real
LangChain tool classes.

### 2. Agent Execution Loop Through Sentinel

**File:** `tests/sentinel/test_integration_langchain.py::TestAgentLoopIntegration`

We built a `SimulatedReActAgent` that mirrors how `langchain.agents.create_react_agent`
works — receiving a task, planning tool calls, executing them through adapters,
and collecting results. This tests the full agent-sentinel integration path.

Scenarios tested:
- Agent searches then calculates — both calls tracked correctly
- Agent calls a failing tool — error captured, subsequent tools still work
- Agent tool call arguments are accurately recorded in trace

**Result:** 3/3 tests pass. Agent loops work correctly through sentinel.

### 3. Chaos Injection Through Real Tools

**File:** `tests/sentinel/test_integration_langchain.py::TestChaosWithRealTools`

The commercial differentiator — chaos injection — tested through real
LangChain tool adapters:

- **Tool failure mid-run:** 100% failure probability on search, agent
  continues to calculator. Sentinel captures the partial degradation.
- **Rate limit errors:** Chaos injector triggers RateLimitError through
  real adapter. Error message and status code captured in trace.
- **Timeout errors:** Timeout injected through real tool. Agent sees
  the error, trace records it.

**Result:** 3/3 tests pass. Chaos injection works through real tools.

### 4. Environment Builder With Real Tools

**File:** `tests/sentinel/test_integration_langchain.py::TestEnvironmentWithRealTools`

`EnvironmentBuilder` creates mock environments that work with real
LangChain tool names and types:

- Builder creates mocks matching LangChain tool names
- Environment with both tools and API for complex agents

**Result:** 2/2 tests pass. Environment builder integrates with real tools.

### 5. Assertions Against Real Traces

**File:** `tests/sentinel/test_integration_langchain.py::TestAssertionIntegration`

Sentinel's 20+ assertions verified against traces generated by real
LangChain tools:

- `assert_tool_called` — detects when a specific tool was/wasn't called
- `assert_no_tool_errors` — verifies clean execution
- `assert_tool_call_order` — validates sequential tool call patterns

**Result:** 3/3 tests pass. Assertions work against real tool traces.

## Running the Integration Tests

pip install -e ".[langchain]"

# Run integration tests
pytest tests/sentinel/test_integration_langchain.py -v

## What This Proves

Before this integration testing:
- "Sentinel works with LangChain" (claimed, unverified)
- "Chaos injection catches failures" (mock-on-mock only)

After this integration testing:
- **Sentinel correctly wraps real LangChain BaseTool instances** — metadata
  preserved, execution delegated to mocks, trace captured
- **Agent execution loops work through sentinel** — tool calls tracked in
  order with arguments and results
- **Chaos injection works through real tool adapters** — failures detected
  and captured in trace, partial degradation visible
- **Assertions catch behavioral regressions** — tool not called, wrong
  order, errors injected, all detectable

## What's Still Needed (Phase 1 Remaining)

The integration tests prove the adapter layer works. For full "proof of value"
we still need:

1. **Real LLM integration** — Test with an actual LangChain agent using a
   real (or mocked) LLM that does tool calling. The current tests simulate
   the agent loop; a real agent would test the full stack.

2. **CrewAI integration** — Similar tests for the CrewAI adapter

3. **OpenAI SDK integration** — Similar tests for the OpenAI adapter

4. **End-to-end scenario** — A YAML scenario that loads, runs a real agent,
   and produces a regression report

## Architecture Decision: Why Simulated Agent Loop?

We chose to simulate the ReAct agent loop rather than using a real LLM
because:

1. **No API key required** — Tests run anywhere, no cost
2. **Deterministic** — Same plan every time, no LLM variance
3. **Fast** — No network calls, tests complete in <0.2s
4. **Focused** — Tests the adapter layer, not LangChain's agent loop

The real LLM integration test is a separate concern — it tests that the
entire stack works end-to-end, including LLM tool calling. That's Phase 1
stretch goal, not core proof of value.
